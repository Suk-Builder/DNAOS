**日期**: 2026-06-12
**章节**: 第 2 章 (进程) + 第 4 章 (调度) + 第 13-15 章 (内存虚拟化)
**目标**: 把握 OS 三大主题 — 虚拟化、并发、持久化的"虚拟化"部分

---

## 第 2 章: 进程 (Process)

### 这章讲的 3 件事
1. **进程 = 一个运行中的程序**。操作系统给每个进程一个"幻觉"——它独占整个 CPU 和内存。
2. **进程状态机**: 3 个状态 — running / ready / blocked。OS 调度器不停切换。
3. **进程控制块 (PCB)**: 操作系统为每个进程存一张表, 记录寄存器值、内存映射、打开的文件等。**切换进程就是换 PCB**。

### DNAOS 对应
- 我们 v20 entry_32 跑完就 halt, **还没有进程概念**。
- **M13 (多任务)** 才需要进程, 这章是 M13 基础。
- 现阶段 DNAOS 是"单进程 OS", 整个内核 = 一个进程。

---

## 第 4 章: 调度 (Scheduling)

### 这章讲的 3 件事
1. **FIFO (先来先服务)**: 简单, 但"护航效应" (一个长任务占 CPU, 后面的都得等)。
2. **最短任务优先 (SJF)**: 平均周转时间最优, 但**长任务可能饿死**。
3. **轮转 (Round Robin)**: 给每个任务固定时间片, 时间片到了就切换。**响应时间好, 但周转时间差**。
4. **多级反馈队列 (MLFQ)**: 现代 OS 都用这个。结合"优先级 + 时间片"。

### DNAOS 对应
- **M13 之前都不用**。我们的 v20 halt 是"非抢占"调度, 实际就是"单任务跑到底"。
- 但**M11 (长模式 + 64-bit) 之后**如果要做交互式 shell, RR 调度就是基础。
- MLFQ 太复杂, 我们的 M13 用 **简单 RR (10ms 时间片)** 就够。

## 第 13-15 章: 内存虚拟化

### 这章讲的 3 件事

#### 13 章: 地址空间 (Address Space)
- **虚拟地址**: 进程看到的地址, 看起来独占整个内存。
- **物理地址**: 真实 RAM 里的位置。
- OS 维护一个"虚拟 → 物理"的映射表, 每次访问内存都查表。

#### 14 章: 分页 (Paging)
- **页表 (Page Table)**: 把虚拟地址空间切成固定大小的"页" (4KB), 物理内存也切成"页帧"。页表记录"虚拟页 X → 物理页帧 Y"。
- **页表项 (PTE)**: 包含 物理页帧号 + 有效位 + 读/写/执行权限位。
- **页表基址寄存器 (CR3)**: CPU 知道当前进程的页表在哪。
- **TLB** (Translation Lookaside Buffer): CPU 内部的页表缓存, 命中就快, 没命中就"TLB miss" → 走页表 (慢)。

#### 15 章: 分页的细节
- **页大小**: x86 默认 4KB。Linux 也支持 2MB、1GB 大页。
- **多级页表**: 一级页表太大装不下 → 二级、三级、四级 (x86-64 用 4 级)。
- **页面置换算法**: 当物理内存不够, 把不常用的页换到磁盘。LRU、Clock、Working Set 等。

### DNAOS 对应 ⭐

**M12 (内存分页) 直接抄**。具体要做的:
1. 启用 CR0.PG (CR0 bit 31) — **关键是 CR3 要指向页目录**
2. 页目录放哪? **Linus 的 head.s 把它放在地址 0x00000** (`_pg_dir:`, 4KB 大小, 1024 个 PTE)
3. 我们 8MB flat model: 1024 个 PTE, 每个 PTE 指向 4KB, 总 4MB. 再加 1024 个 PTE → 8MB
4. **Linus 用了 3 个页目录项**: pg0+7, pg1+7, pg1+4092 (后面是 reverse-fill 的 trick)

**写 M12 时, 把 Linus head.s 的 setup_paging 段**完整抄过来 + 改大小。
